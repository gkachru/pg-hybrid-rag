# Arabic Dialectal RAG Benchmark

Measures **Arabic retrieval quality** over real banking and telecom help-center FAQ content with dialect robustness as the headline axis. Every query exists in three Arabic variants — MSA, Saudi, and Moroccan Darija — all sharing one target FAQ answer document, so metrics (Recall@k, MRR@10, nDCG@10) slice by dialect and domain. SAMA/UAE T&C PDFs are chunked in as in-domain distractors.

The benchmark mirrors `examples/playground.ts`: it creates a fresh isolated Postgres database per config, migrates, indexes the corpus, runs all queries, scores results, then drops the database.

Same requirements as the playground:

- **`.env` at repo root** with the following variables:

## Corpus and Queries

### What is committed

- `examples/benchmark/queries.json` — authored query set: MSA/Saudi/Darija variants per query, `target_doc` id, domain, provider. MSA variants are the real FAQ questions verbatim; dialect variants are authored rewrites.
- `examples/benchmark/scrape-notes.md` — provenance of all FAQ scrapes (sources, dates, method, pair counts).
- `datasets/PDFs/` — SAMA/UAE T&C PDFs (distractor corpus source). Already present in the repo.

### What is gitignored (not redistributed)

- `datasets/faqs/*.jsonl` — scraped FAQ snapshots (Zain Kuwait, ABK, SAB).
- `datasets/benchmark-cache/` — extracted PDF text and built corpus cache.
- `examples/benchmark/results/` — timestamped result JSON files.

These artifacts are local-only. The FAQ snapshots must be re-created by scraping (see `scrape-notes.md`); the extracted/corpus files are regenerated by the prep steps below.

### Corpus disclosure

The corpus is **terms/policy documents (PDF distractors) + FAQ help-center content**. Scored targets are FAQ answer documents; PDFs contribute only as retrieval noise. Queries are Arabic in three dialects (MSA, Saudi, Darija) each targeting one FAQ answer document. MSA retrieves best; dialect variants degrade — this degradation is the intended signal showing how well the dense vector leg carries cross-dialect retrieval.

## Prep (one-time)

Run these steps once to build the local corpus artifacts. They are idempotent.

### Step 1 — Extract PDFs

Extracts raw per-page text from `datasets/PDFs/` using pypdf. Cleaning (boilerplate strip + Arabic-script filter) happens in `cleanArabic.ts` at corpus-build time; this step only extracts raw text.

```bash
# Run from repo root
python examples/benchmark/extract_pdfs.py
```

Output: `datasets/benchmark-cache/extracted.jsonl`

### Step 2 — Ensure FAQ snapshots exist

The FAQ JSONL files (`datasets/faqs/*.jsonl`) are scraped help-center content and are gitignored. They are not an automatic step — re-create them by scraping the sources documented in `examples/benchmark/scrape-notes.md`. The committed `queries.json` references their `doc_id` values, so the corpus must contain those documents for queries to resolve.

### Step 3 — Build the corpus

Chunks FAQ answers (scored targets) and Arabic-filtered PDF text (distractors) using the library's `Chunker` with `tokenLimit: 512`.

```bash
bun run examples/benchmark/buildCorpus.ts
```

Output: `datasets/benchmark-cache/corpus.jsonl`

**Escape hatch:** pass `--include-question-in-doc` to also index the FAQ question alongside the answer (default is answer-only, which is a stricter retrieval test):

```bash
bun run examples/benchmark/buildCorpus.ts --include-question-in-doc
```

The corpus is loaded from cache on subsequent runs; delete `datasets/benchmark-cache/corpus.jsonl` to force a rebuild.

## Running the Benchmark

### Single config (baseline)

```bash
bun run examples/benchmark/run.ts
```

### Single config with flags

Same flags as the playground:

```bash
bun run examples/benchmark/run.ts --bm25
bun run examples/benchmark/run.ts --bm25 --rerank
bun run examples/benchmark/run.ts --bm25 --vectorchord --rerank --cjk
```

### Matrix sweep (5 presets)

Runs five configs in sequence — `baseline`, `+bm25`, `+vectorchord`, `+rerank`, `all` — each on its own freshly-migrated isolated database, then prints a cross-config comparison table:

```bash
bun run examples/benchmark/run.ts --matrix
```

`--cjk` applies to all matrix configs when combined:

```bash
bun run examples/benchmark/run.ts --matrix --cjk
```

### Additional flags

| Flag | Default | Description |
|---|---|---|
| `--topk N` | `10` | Retrieval depth (Recall@k computed up to this k) |
| `--limit-queries N` | all | Run only the first N resolved queries (smoke test) |
| `--judge` | off | Opt-in LLM-judge pass over ~15 MSA queries on the last config's live DB |

**Smoke run example:**

```bash
bun run examples/benchmark/run.ts --limit-queries 6 --topk 10
```

## Scoring

**Ground truth:** a retrieval result is relevant if and only if its source document matches the query's `target_doc` (the FAQ answer document the query was written to retrieve).

**Metrics computed:** Recall@1, Recall@3, Recall@5, Recall@10, MRR@10, nDCG@10.

**Slices:** overall, by dialect (`msa` / `saudi` / `darija`), by domain (`banking` / `telecom`).

**Unresolved queries:** queries whose `target_doc` is not present in the built corpus (e.g. the FAQ snapshot is missing) are reported and excluded from scoring.

**PDF distractors:** PDF chunks are indexed as noise; no query targets them. Their presence makes precision meaningful.

**LLM judge (`--judge`):** optional open-ended sanity check. Runs on the first ~15 resolved MSA queries using the last config's pipeline. Reports mean precision@k (fraction of top-k results scored ≥ 1 by the judge). Requires `JUDGE_BASE_URL`, `JUDGE_MODEL`, and `JUDGE_API_KEY`.
